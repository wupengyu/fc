# 微信报单数据流程与筛选规则

更新时间：2026-05-08

本文档说明当前项目从 Redis 接收微信消息，到原始消息入库、报单筛选、AI 解析、订单明细、号码统计、查询展示、数据审计的完整链路。本文以当前代码为准，旧千寻 HTTP 回调、SSE、千寻事件处理、手工 raw 报单入口均已停用或移除。

## 1. 当前有效入口

| 通道 | 当前状态 | 入口代码 | 说明 |
| --- | --- | --- | --- |
| Redis 微信消息队列 | 启用 | `RedisClientService` | 当前唯一正式微信消息来源，队列名 `wechat_messages` |
| Redis processing 队列 | 启用 | `RedisClientService` | 队列名 `wechat_messages:processing`，用于可靠消费和启动恢复 |
| HTTP `/callback` | 永久停用 | `DisabledCallBackController` | 固定返回 410，不会进入消息处理链路 |
| 旧回调控制器 | 不注册 | `CallBackController` | 已从 Spring Controller 中移除，仅保留空壳说明 |
| 旧千寻事件处理 | 已移除 | 已删除 | 不再有 `RecvMsgGroupEvent`、`RecvMsgFriendEvent` 等事件 Bean |
| SSE 消费 | 已移除 | 已删除 | 不再有 `SseClientService` |
| 手工 API 报单 | 停用 | `OrderController` | `/wechat/api/orders/raw` 固定返回 410 |
| 千寻主动 API | 已移除 | 已删除 | 不再有 `QianXunApi` |

## 2. 总体数据流

| 顺序 | 环节 | 代码 | 数据载体/表 | 输出 |
| --- | --- | --- | --- | --- |
| 1 | Redis 可靠取消息 | `RedisClientService.consumeLoop` | `wechat_messages` -> `wechat_messages:processing` | 单条原始 JSON |
| 2 | 微信消息基础筛选 | `RedisClientService.processMessage` | `SseMessageVo` | 目标群文本消息 |
| 3 | 原始微信消息入库 | `MessageIngressService` | `t_message` | `WECHAT_REDIS` 原始消息 |
| 4 | 报单时间窗口判断 | `OrderWindowService` | 内存 | 窗口内订单消息 |
| 5 | 订单缓冲 | `OrderBufferService` | 内存 buffer | 待刷写订单消息 |
| 6 | 订单刷写 | `OrderFlushTask` | 每 1 秒调度 | raw 入库请求 |
| 7 | 原始报单入库 | `OrderIngressService` | `t_order_raw` | raw 报单 |
| 8 | 异步解析提交 | `AsyncParseService` | `parseTaskExecutor` | AI 解析任务 |
| 9 | 本地快路径或 AI 解析 | `AiParseService` | AI 结果对象 | 成功/跳过/失败结果 |
| 10 | 解析批次入库 | `NormalizePersistService` | `t_order_parse_batch` | 最新 batch |
| 11 | AI 结果入库 | `NormalizePersistService` | `t_ai_parse_result` | 结构化解析记录 |
| 12 | 有效订单明细入库 | `NormalizePersistService` | `t_order_item` | 订单项 |
| 13 | 号码明细入库 | `NormalizePersistService` | `t_order_item_number` | 每个号码一行 |
| 14 | 号码统计汇总 | `NumberStatsMapper` | `t_number_stats` | 按号码汇总注数/金额 |
| 15 | 查询与审计 | `StatsController`、`DataAuditService` | 统计表和明细表 | 页面/API/审计结果 |

## 3. Redis 消息消费与基础筛选

入口：`RedisClientService.processMessage`

| 筛选顺序 | 判断条件 | 通过后动作 | 不通过动作 |
| --- | --- | --- | --- |
| 1 | 从 `wechat_messages` 取出并移入 `wechat_messages:processing` | 进入 JSON 解析 | 无消息则阻塞等待 |
| 2 | JSON 能解析成 `SseMessageVo` | 继续 | 记为丢弃并 ack processing |
| 3 | `isGroup == true` | 继续 | 丢弃并 ack |
| 4 | `username == target-group-wxid` | 继续 | 丢弃并 ack |
| 5 | `type == "文本"` | 继续 | 丢弃并 ack |
| 6 | 构造 `msgId` | 继续 | 缺少上游 id 不丢弃 |
| 7 | 构造 `RecvMsgReqVo` 并写入 `t_message` | 成功/重复均继续订单窗口判断 | 入库失败则 requeue 回主队列 |
| 8 | `receivedAt` 是否在报单窗口 | 加入订单缓冲 | 窗口外只保留 `t_message`，不进 raw |
| 9 | 本条处理完成 | 从 processing ack 删除 | 异常则 requeue 回主队列 |

Redis `msgId` 生成优先级：

| 优先级 | 字段 | 生成种子 |
| --- | --- | --- |
| 1 | `localId` | `REDIS|LOCAL|群ID|localId` |
| 2 | `sortSeq` | `REDIS|SORT|群ID|sortSeq` |
| 3 | `serverId` | `REDIS|SERVER|群ID|serverId` |
| 4 | 都没有 | `REDIS|FALLBACK|群ID|发送人|时间戳|内容|UUID` |

关键规则：

| 规则 | 说明 |
| --- | --- |
| 可靠消费 | 使用 `rightPopAndLeftPush`，先进入 processing，处理成功后 ack |
| 启动恢复 | 服务启动时把 processing 队列残留搬回主队列 |
| 缺少上游 id 不按内容去重 | fallback 加 UUID，避免相同文本重复报单被吞 |
| `t_message` 重复不阻断 raw 检查 | Redis 重放时，如果原始消息已入库，仍会继续检查 raw 是否需要解析 |
| 写库失败不丢消息 | `MessageIngressService` 返回 `FAILED` 时 requeue |
| `receivedAt` 来自消息时间戳 | 秒级时间戳自动乘 1000 转毫秒 |

## 4. `t_message` 原始微信消息入库与去重

入口：`MessageIngressService`

| 项目 | 规则 |
| --- | --- |
| 入库表 | `t_message` |
| 唯一有效 source | `WECHAT_REDIS` |
| 非 Redis source | 直接 `REJECTED`，不入库 |
| fingerprint | 有 `msgId` 时为 `source|MSG_ID|msgId` 的 SHA-256 |
| 无 msgId fingerprint | `source|NO_ID|timestamp|fromWxid|senderWxid|UUID` 的 SHA-256 |
| 去重范围 | 同一个 `source` 内判断 |
| 同时有 msgId 和 fingerprint | 命中同 source 的相同 `msgId` 或相同 `fingerprint` 即重复 |
| 只有 msgId | 同 source 相同 `msgId` 即重复 |
| 只有 fingerprint | 同 source 相同 `fingerprint` 即重复 |
| DB 兜底 | 启动时尽量创建 `uk_t_message_source_msg_id`、`uk_t_message_source_fingerprint` |
| 写入失败 | 返回 `FAILED`，Redis 消费端把 payload requeue 回主队列 |

这里的去重目标只是不让同一条微信传输消息重复入库，不用于判断业务报单是否重复。业务上，同样文本重复发出来仍可能是有效报单。

## 5. 报单窗口与缓冲筛选

入口：`OrderWindowService`、`OrderBufferService`、`OrderFlushTask`

| 环节 | 规则 |
| --- | --- |
| 报单窗口 | `19:30:00-21:35:00`，含开始和结束时间 |
| 第一次窗口判断 | Redis 消息入 `t_message` 后判断，窗口内才进入订单缓冲 |
| 缓冲触发 | 缓冲数量达到 `order.trigger-count=50` 或等待超过 `order.trigger-wait=500ms` |
| 调度频率 | `OrderFlushTask` 每 1 秒检查一次 |
| 第二次窗口判断 | 刷写前再次过滤窗口外消息 |
| 窗口外消息 | 保留在 `t_message`，但不进入 `t_order_raw` |
| raw 入库前失败 | 放回订单缓冲等待下次处理 |
| raw 入库已尝试后失败 | 不盲目回塞，避免造成重复 raw 风险 |

## 6. `t_order_raw` 原始报单入库筛选

入口：`OrderIngressService.batchIngest`

| 筛选顺序 | 规则 | 目的 |
| --- | --- | --- |
| 1 | source 为空时默认 `WECHAT`，非空转大写 | 统一来源格式 |
| 2 | 只允许 `WECHAT_REDIS` | Redis 是唯一有效来源 |
| 3 | 生成 raw fingerprint | 作为 raw 级消息身份 |
| 4 | 同一批次内相同 fingerprint 跳过 | 防止同一传输消息重复提交 |
| 5 | 批量预查 DB 中已存在 fingerprint | 减少逐条查询 |
| 6 | 进入 `ingestLock` 同步区 | 防止高并发竞态 |
| 7 | DB 中 fingerprint 已存在则跳过新增 raw | 防止传输重复入库 |
| 8 | 已存在 raw 但无任何 parse batch | 返回 existing raw 继续解析 |
| 9 | 插入 `t_order_raw` | 进入 AI 解析 |
| 10 | DB 唯一键异常兜底 | 并发下最后一刻重复则跳过，并检查是否需要续跑 |

raw fingerprint 生成规则：

| 情况 | seed |
| --- | --- |
| 有 `msgId` | `source|msgId` |
| 无 `msgId` | `source|fromWxid|senderWxid|UUID` |

已移除的旧规则：

| 旧规则 | 当前状态 | 原因 |
| --- | --- | --- |
| 跨来源镜像重复判断 | 已移除 | 当前只有 Redis 来源，不再需要 Redis/SSE/回调之间互相压制 |
| 10 分钟 recent identity cache | 已移除 | DB fingerprint 和同批 fingerprint 已可覆盖传输重复 |
| 按 rawText 内容去重 | 不使用 | 相同文本可能是有效重复报单 |
| 旧 `WECHAT`、`WECHAT_CALLBACK`、`WECHAT_SSE` 新增 raw | 不允许 | 防止旧通道混入 |

重要业务边界：

| 场景 | 处理方式 |
| --- | --- |
| 同一上游 `msgId` 重复推送 | 视为传输重复，跳过 |
| 不同消息 id 但文本相同 | 视为不同微信消息，原则上保留并解析 |
| 重复有效报单 | 不能仅因内容相同就丢弃 |
| 旧历史 `WECHAT` raw | 保留历史数据；新代码不再新增，也不会作为当前有效来源 |
| Redis 重放断点 | raw 已存在但无 batch 时会继续解析 existing raw |

## 7. AI 解析筛选与纠偏

入口：`AsyncParseService`、`AiParseService`

| 环节 | 规则 |
| --- | --- |
| 解析方式 | raw 批次提交异步线程，AI 内部并行处理每条 raw |
| 实时并行 | `ai.realtime-parallel=16` |
| 重跑并行 | `ai.reparse-parallel=2` |
| AI 超时 | `ai.timeout=180000ms` |
| AI 重试 | 对 timeout、connection reset/refused、502、503、504 等重试 |
| AI call log | `ai.save-call-log=false` 时不保存调用日志 |

本地快路径优先于 AI API：

| 顺序 | 快路径 | 用途 |
| --- | --- | --- |
| 1 | exact prompt case | 命中历史人工纠偏精确案例 |
| 2 | simple direct list | 简单直选列表 |
| 3 | long half direct list | 长列表半数/金额类直选 |
| 4 | long direct list | 长直选列表 |
| 5 | annotated long permutation list | 带源注数标注的长复式/排列列表 |
| 6 | long permutation list | 长复式/排列列表 |

AI 提示词增强：

| 来源 | 代码 | 作用 |
| --- | --- | --- |
| 基础提示词 | `PromptService` | 彩票报单解析规则 |
| 目标群规则 | `TargetGroupRuleService` | “福停”后强制按体彩排列三处理 |
| 常见错字归一化 | `OrderTextNormalizationService` | 复试->复式、丹->单、租->组等 |
| 历史纠偏案例 | `PromptCaseService` | 将人工修正案例加入提示词或精确命中 |

AI 响应筛选：

| 条件 | 结果 |
| --- | --- |
| JSON 数组解析成功 | 进入结果纠偏 |
| JSON 为空 | 生成 `FAIL`，原因“AI响应中未找到任何结果” |
| JSON 解析异常 | 生成 `FAIL`，原因“响应解析失败” |
| AI 调用异常 | 生成 `FAIL`，记录异常信息 |
| `valid=true/status=SUCCESS` 且有 `data` | 作为成功候选 |
| `status=SKIP` | 作为跳过候选，不入订单明细 |
| `status=FAIL` | 作为失败候选，不入订单明细 |

主要后处理筛选/纠偏：

| 纠偏类型 | 作用 |
| --- | --- |
| exact case correction | 精确命中人工案例时优先采用案例结果 |
| 多行包票纠偏 | 按行解析复杂包票，防止 AI 合并错行 |
| 彩种作用域纠偏 | 多行、多段、空行中彩种只影响对应范围 |
| 直选/复式混合纠偏 | 一条消息同时存在直选和复式时分开处理 |
| 复式展开纠偏 | 将基础号码展开成全排列号码 |
| 共享注数纠偏 | 多行共用一个注数时补齐 |
| 金额保护 | 纠偏后总金额异常时回退可信基础结果 |
| 号码注数对纠偏 | 识别“号码 注数”成对结构 |
| 豹子/组选包票纠偏 | 特殊玩法结构化处理 |
| 两码跳过 | 仅两位号码且属于两码/二码玩法时标记 `SKIP` |
| 不支持组选跳过 | 明确属于暂不支持类型时标记 `SKIP` |
| 无明确三位数防误算 | 对无明确三位号码且不属于豹子/组选等特殊结构的结果做保护 |
| 福停后强制体彩 | 目标群出现“福停”后，后续消息强制 `TC/P3` |

## 8. 解析结果落库筛选

入口：`NormalizePersistService.persistSingle`

| 顺序 | 规则 | 落表 |
| --- | --- | --- |
| 1 | 对 raw 执行 `selectByIdForUpdate` | 防止同一 raw 并发落库 |
| 2 | AI items 为空 | 保存失败 batch |
| 3 | 有成功 item | 如已有解析数据，先清理该 raw 历史解析和统计 |
| 4 | 无成功但有失败 | 保存失败 batch |
| 5 | 无成功但有跳过 | 保存跳过 batch |
| 6 | 保存 batch | `t_order_parse_batch` |
| 7 | 成功 batch 设 `is_effective=1` | 其他同 raw batch 置 `is_effective=0` |
| 8 | 每个 AI item 都保存记录 | `t_ai_parse_result` |
| 9 | 只有 success item 进入订单明细 | `t_order_item`、`t_order_item_number` |

batch 状态：

| 状态 | 数值 | 是否有效 | 是否进入统计 |
| --- | --- | --- | --- |
| 成功 | `parse_status=1` | `is_effective=1` | 是 |
| 部分成功 | `parse_status=2` | 当前常量保留 | 视具体实现 |
| 失败 | `parse_status=3` | `is_effective=0` | 否 |
| 跳过 | `parse_status=4` | `is_effective=0` | 否 |

## 9. 订单明细与号码统计筛选

入口：`NormalizePersistService.persistValidOrders`

| 筛选顺序 | 规则 | 结果 |
| --- | --- | --- |
| 1 | success item 的 `data` 为空 | 跳过该 item |
| 2 | `numbers` 为空 | 跳过该 item |
| 3 | 去掉 null 号码后为空 | 跳过该 item |
| 4 | `play` 为空 | 默认 `直选` |
| 5 | `zone` 为空 | 默认 `MAIN` |
| 6 | `bet` 非正数或为空 | 默认 `1` |
| 7 | `amount` 大于 0 | 按号码数量平均分摊到每个号码 |
| 8 | 同 raw 内贡献键重复 | 跳过重复贡献 |
| 9 | 有效号码为空 | 不生成 `order_item` |
| 10 | 批量保存订单项 | `t_order_item` |
| 11 | 批量保存号码明细 | `t_order_item_number` |
| 12 | 按统计键汇总后批量 upsert | 累加 `t_number_stats` |

同 raw 内统计防重键：

| 字段 | 说明 |
| --- | --- |
| `lotteryCategory` | 彩种 |
| `gameType` | 游戏 |
| `playType` | 玩法 |
| `numberZone` | 区域 |
| `number` | 号码 |
| `betCount` | 注数 |

该规则的目的不是删除重复微信报单，而是防止同一条 raw 被 AI 重复输出同一号码贡献，导致同一 raw 内重复加统计。

统计写入保护：

| 机制 | 说明 |
| --- | --- |
| `stat_applied` | 新落库路径在同一事务内写入 `1`，表示本 item 已随本次统计一起应用 |
| 同 raw 贡献键 | 入库前先去掉同 raw 内 AI 重复输出的同号码贡献 |
| 批量统计 upsert | 所有号码贡献先在内存按统计键汇总，再批量写入 `t_number_stats` |
| 事务保护 | 订单项、号码明细、统计任一失败会整体回滚 |
| `ON DUPLICATE KEY UPDATE` | 同号码维度存在则累加，不存在则插入 |

## 10. 清理、重跑与统计重建规则

入口：`OrderReparseService`、`NormalizePersistService`、`StatsRebuildService`

| 通道 | 是否自动 | 时间限制 | 说明 |
| --- | --- | --- | --- |
| 自动补偿重跑 | 当前关闭 | 受窗口限制 | `order.auto-reparse-enabled=false` |
| `/api/reparse` | 手动 | 受窗口限制 | 普通手动按日期重跑 |
| `/api/force-parse` | 手动 | 不受窗口限制 | 强制按日期重跑 |
| `/api/force-parse-fast` | 手动 | 收单窗口内默认禁用 | 使用实时 lane 快速重跑 |
| `/api/force-parse-raw` | 手动 | 不受窗口限制 | 单 raw 重跑 |
| `/api/reparse-corrected-raw` | 手动 | 不受窗口限制 | 带人工校正说明重跑 |
| `/api/rebuild-stats` | 手动 | 不受限制 | 从订单明细重建指定日期 `t_number_stats` |

重跑候选筛选：

| 条件 | 规则 |
| --- | --- |
| 成功 batch | 不重跑 |
| 失败 batch | 只有 parseMsg 像 AI API、timeout、无法解析等可重试错误才重跑 |
| 无 batch | 可能实时解析仍在路上，需满足更长冷却 |
| 重试次数 | 达到 `order.reparse-max-attempts` 后跳过 |
| 冷却时间 | 默认 `order.reparse-cooldown-seconds` |
| 批大小 | 最多 `order.reparse-batch-size` |
| 并发保护 | `retryRunning` 防止多个补偿任务同时跑 |

清理逻辑：

| 清理动作 | 说明 |
| --- | --- |
| 锁定 raw | `selectByIdForUpdate` 防止并发清理/落库 |
| 查旧 batch | 包含历史 batch id |
| 查旧 item | 按 raw_id 和 batch_id 双路径查，兜底错挂明细 |
| 回滚统计 | 仅对 `stat_applied=1` 的 item 扣减统计 |
| 删除号码明细 | 删除 `t_order_item_number` |
| 删除订单项 | 删除 `t_order_item` |
| 删除 parse batch | 删除 `t_order_parse_batch` |
| 删除 AI 结果 | 删除 `t_ai_parse_result` |
| 日期清理 | 清理指定日期解析结果，保留 raw |
| 删除非正统计 | 清除 `order_count<=0` 或 `sum_amount<=0` 的统计行 |

统计重建逻辑：

| 步骤 | 说明 |
| --- | --- |
| 1 | 查询重建前 `t_number_stats` 总注数和金额 |
| 2 | 查询明细表重算总注数和金额 |
| 3 | 删除指定 `issue_key` 的 `t_number_stats` |
| 4 | 从 `t_order_item + t_order_item_number + effective success batch` 聚合插入 |
| 5 | 再次对比统计表和明细表，返回 `statsMismatchCount` |

## 12. 核查用 SQL 思路

以下是排查数据链路时常用的核查方向，实际执行时按目标日期替换 `:date`。

| 目标 | 核查点 |
| --- | --- |
| Redis 是否积压 | `LLEN wechat_messages`、`LLEN wechat_messages:processing` 或 `/wechat/api/runtime-status` |
| raw 是否都解析 | Redis source 的 `t_order_raw left join t_order_parse_batch` 找无 batch raw |
| 最新状态分布 | 每个 raw 取最新 batch，按 source 和 `parse_status` 分组 |
| 是否有失败 | Redis source 最新 batch `parse_status=3` |
| 统计是否一致 | 从 `t_order_item + t_order_item_number + effective batch` 重算，对比 `t_number_stats` |
| 是否有无效 batch 入统计 | `t_order_item` 关联 batch 非成功或非 effective |
| 是否有空号码成功项 | 成功有效 item 但无 `t_order_item_number` |
| raw 层重复 | 按 source/from/sender/rawText/receivedAt 分组 |
| 近似重复 | 同 source/from/sender/rawText 在 60 秒内成对出现 |
| 旧来源残留 | `t_message`、`t_order_raw` 按 source 分组 |

raw 层重复不一定是错误。判断原则是：

| 类型 | 是否错误 | 处理 |
| --- | --- | --- |
| 同一上游 msgId 重复 | 通常是传输重复 | 应被 `t_message` 或 `t_order_raw` 去重 |
| 不同上游消息但文本相同 | 可能是有效重复报单 | 不应按内容硬删除 |
| 旧 WECHAT 与 WECHAT_REDIS 历史镜像 | 历史遗留，不代表新链路异常 | 新数据只看 Redis source |
| 同 raw 内 AI 重复输出 | 通常是解析冗余 | 由贡献键和 `stat_applied` 防重 |

## 13. 当前重点保护规则

| 风险 | 当前保护 |
| --- | --- |
| 旧千寻回调混入 | `/callback` 固定 410，旧事件 Bean 和千寻 API 已删除 |
| 手工 raw 混入 | `/api/orders/raw` 固定 410 |
| Redis 消息处理中宕机 | processing 队列 + 启动恢复 |
| Redis 高峰积压 | 4 个 Redis 消费线程，订单缓冲 50 条或 500ms 即刷写，实时 AI 并行 16 |
| 自动重跑抢占高峰资源 | 自动补偿重跑关闭，快速重跑窗口内默认禁用 |
| 相同文本有效报单被吞 | 无上游 id 时引入 UUID，不按内容生成唯一身份 |
| 跨来源历史数据压制 Redis | 新 raw 只允许 `WECHAT_REDIS`，已移除跨来源镜像去重 |
| AI 重复输出同号码 | 同 raw 内贡献键防重，`stat_applied` 防重复统计 |
| 单 raw 重跑重复入账 | 重跑前先清理旧明细并扣减旧统计 |
| 统计表漂移 | `/api/audit-date` 可对账，`/api/rebuild-stats` 可按明细重建 |
| 长复式/排列单 AI 超时 | 本地长列表快路径优先处理 |

## 14. 业务排障路径

| 问题 | 优先检查 |
| --- | --- |
| 今天是否还在收消息 | `/wechat/api/runtime-status`、`/wechat/api/check-messages`、Redis 队列长度、`t_message` 最新时间 |
| 消息是否进入 raw | `t_message` 与 `t_order_raw` 按时间、sender、内容对比，重点看 `WECHAT_REDIS` |
| raw 是否漏解析 | `/wechat/api/audit-date` 的 `redisRawWithoutAnyBatch` |
| 解析是否失败 | `/wechat/api/audit-date` 的 `latestParseStatusBySource` 和 `warnings` |
| 注数偏高 | 查 `t_order_item`、`t_order_item_number`、同 raw 贡献重复、raw 层重复样例 |
| 某号码来源 | `/wechat/api/top-source-messages?betNumber=xxx` |
| 某 raw 明细 | 查 `t_order_parse_batch`、`t_ai_parse_result`、`t_order_item`、`t_order_item_number` |
| 统计不一致 | `/wechat/api/audit-date` 看 `statsMismatchCount`，必要时 `/wechat/api/rebuild-stats` |
| 旧通道影响 | 审计中的非 Redis source 只作为历史残留关注，新链路不再接收 |

推荐日常检查顺序：

| 顺序 | 动作 |
| --- | --- |
| 1 | 看 `/wechat/api/runtime-status`，确认 Redis 主队列、processing、orderBuffer、失败数 |
| 2 | 看 `/wechat/api/audit-date?date=当天日期`，确认 `redisRawWithoutAnyBatch=0`、`statsMismatchCount=0` |
| 3 | 如统计不一致，先确认没有实时任务在跑，再执行 `/wechat/api/rebuild-stats?date=当天日期` |
| 4 | 如某号码异常，用 `top-source-messages` 找 raw 原文逐条核对 |
