# 微信报单数据流程与筛选规则

更新时间：2026-05-06

本文档说明当前项目从微信消息接收到 AI 解析、订单明细、号码统计、查询展示的完整数据流转链路，并列出每一层筛选、去重、跳过、重跑和统计防重规则。

## 1. 当前有效入口

| 通道 | 当前状态 | 入口代码 | 说明 |
| --- | --- | --- | --- |
| Redis 微信消息队列 | 启用 | `RedisClientService` | 当前唯一正式微信消息来源，队列名 `wechat_messages` |
| 千寻 HTTP 回调 `/callback` | 停用 | `DisabledCallBackController` | `wechat.callback.enabled=false` 时返回 410，不再接收微信消息 |
| 旧千寻事件处理 | 条件启用 | `CallBackController`、`RecvMsgGroupEvent` | 只有 `wechat.callback.enabled=true` 才会启用 |
| SSE | 停用 | `SseClientService` | `sse.enabled=false`，保留代码但不启动消费线程 |
| 手工 API 报单 | 保留 | `OrderController` | `/wechat/api/orders/raw`，用于人工或测试提交 |
| 千寻主动 API | 停用 | `QianXunApi` | `qian-xun-pro.enabled=false` 时不装配 |

当前配置位于 `src/main/resources/application.yml`：

| 配置 | 当前值 | 作用 |
| --- | --- | --- |
| `server.port` | `8989` | 服务端口 |
| `server.servlet.context-path` | `/wechat` | 所有接口前缀 |
| `redis.queue-name` | `wechat_messages` | Redis 消费队列 |
| `target-group-wxid` | `49792312233@chatroom` | 只处理该目标群 |
| `wechat.callback.enabled` | `false` | 关闭旧 HTTP 回调 |
| `sse.enabled` | `false` | 关闭 SSE 通道 |
| `qian-xun-pro.enabled` | `false` | 关闭千寻主动 API |
| `order.window-start` | `19:30:00` | 报单开始时间 |
| `order.window-end` | `21:35:00` | 报单结束时间 |
| `order.trigger-count` | `50` | 订单缓冲满 50 条触发刷写 |
| `order.trigger-wait` | `500` | 第一条进入缓冲后等待 500ms 触发刷写 |
| `ai.realtime-parallel` | `16` | 实时 AI 解析并行度 |
| `ai.reparse-parallel` | `2` | 重跑通道 AI 解析并行度 |
| `order.auto-reparse-enabled` | `false` | 关闭自动补偿重跑 |

## 2. 总体数据流

| 顺序 | 环节 | 代码 | 数据载体/表 | 输出 |
| --- | --- | --- | --- | --- |
| 1 | Redis 拉取微信消息 | `RedisClientService.consumeLoop` | Redis list `wechat_messages` | 原始 JSON |
| 2 | 微信消息基础筛选 | `RedisClientService.processMessage` | `SseMessageVo` | 目标群文本消息 |
| 3 | 原始微信消息入库 | `MessageIngressService` | `t_message` | 微信原始消息 |
| 4 | 报单时间窗口筛选 | `OrderWindowService` | 内存 | 窗口内订单消息 |
| 5 | 订单缓冲 | `OrderBufferService` | 内存 buffer | 批量刷写任务输入 |
| 6 | 原始报单入库 | `OrderIngressService` | `t_order_raw` | raw 报单 |
| 7 | 异步解析提交 | `AsyncParseService` | AI 线程池 | 单 raw 独立解析任务 |
| 8 | 本地快路径或 AI 解析 | `AiParseService` | AI 结果对象 | 成功/跳过/失败结果 |
| 9 | 解析批次入库 | `NormalizePersistService` | `t_order_parse_batch` | 最新有效 batch |
| 10 | AI 结果入库 | `NormalizePersistService` | `t_ai_parse_result` | 结构化解析记录 |
| 11 | 有效订单明细入库 | `NormalizePersistService` | `t_order_item` | 订单项 |
| 12 | 号码明细入库 | `NormalizePersistService` | `t_order_item_number` | 每个号码一行 |
| 13 | 号码统计汇总 | `NumberStatsMapper` | `t_number_stats` | 按号码汇总注数/金额 |
| 14 | 查询展示 | `StatsController`、`StatsService` | 统计表和明细表 | 页面/API 结果 |

## 3. Redis 消息筛选细节

入口：`RedisClientService.processMessage`

| 筛选顺序 | 判断条件 | 通过后动作 | 不通过动作 |
| --- | --- | --- | --- |
| 1 | Redis 消息 JSON 能解析成 `SseMessageVo` | 继续 | 记录异常，不入库 |
| 2 | `isGroup == true` | 继续 | 直接忽略 |
| 3 | `username == target-group-wxid` | 继续 | 直接忽略 |
| 4 | `type == "文本"` | 继续 | 直接忽略 |
| 5 | 构造 `msgId` | 继续 | 不会因为缺少上游 id 丢弃 |
| 6 | 构造 `RecvMsgReqVo` | 继续 | 无 |
| 7 | 写入 `t_message` | 成功后继续 | 失败进入消息重试缓冲 |
| 8 | 判断是否重复微信消息 | 非重复继续 | 重复则返回，不进入订单缓冲 |
| 9 | `receivedAt` 是否在报单窗口 | 加入订单缓冲 | 窗口外只记录日志，不作为报单 |

Redis `msgId` 生成优先级：

| 优先级 | 字段 | 生成种子 |
| --- | --- | --- |
| 1 | `localId` | `REDIS|LOCAL|群ID|localId` |
| 2 | `sortSeq` | `REDIS|SORT|群ID|sortSeq` |
| 3 | `serverId` | `REDIS|SERVER|群ID|serverId` |
| 4 | 都没有 | `REDIS|FALLBACK|群ID|发送人|时间戳|内容|UUID` |

关键业务规则：

| 规则 | 说明 |
| --- | --- |
| 缺少上游消息 id 时不会按内容去重 | fallback 会加入 UUID，避免同一个人重复发相同报单被吞掉 |
| `sender` 会做轻量身份归一化 | 去掉补充字符、控制字符，降低微信昵称特殊字符影响 |
| `receivedAt` 来自消息时间戳 | 秒级时间戳自动乘 1000 转毫秒 |

## 4. `t_message` 原始微信消息入库与去重

入口：`MessageIngressService`

| 项目 | 规则 |
| --- | --- |
| 入库表 | `t_message` |
| source | Redis 消息固定为 `WECHAT_REDIS` |
| fingerprint | 有 `msgId` 时为 `source|MSG_ID|msgId` 的 SHA-256 |
| 无 msgId fingerprint | `source|NO_ID|timestamp|fromWxid|senderWxid|UUID` 的 SHA-256 |
| 去重范围 | 同一个 `source` 内判断 |
| 同时有 msgId 和 fingerprint | 命中同 source 的相同 `msgId` 或相同 `fingerprint` 即重复 |
| 只有 msgId | 同 source 相同 `msgId` 即重复 |
| 只有 fingerprint | 同 source 相同 `fingerprint` 即重复 |
| DB 兜底 | 启动时尽量创建 `uk_t_message_source_msg_id`、`uk_t_message_source_fingerprint` |
| 写入失败 | 放入 `MessageBufferService`，由 `MessageFlushTask` 每 `message.flush-interval` 重试 |

这里的去重目标是防止“同一条微信传输消息”重复入库，不用于判断“业务报单是否重复”。业务上，同样文本重复发出来可能仍是有效报单。

## 5. 报单窗口与缓冲筛选

入口：`OrderWindowService`、`OrderBufferService`、`OrderFlushTask`

| 环节 | 规则 |
| --- | --- |
| 报单窗口 | `19:30:00-21:35:00`，含开始和结束时间 |
| 第一次窗口判断 | Redis 消息入 `t_message` 后判断，窗口内才进入订单缓冲 |
| 缓冲触发 | 缓冲数量达到 `order.trigger-count=50` 或等待超过 `order.trigger-wait=500ms` |
| 调度频率 | `OrderFlushTask` 每 1 秒检查一次 |
| 第二次窗口判断 | 刷写前再次过滤窗口外消息 |
| 窗口外消息 | 保留在 `t_message`，但不进入 `t_order_raw` |
| 刷写失败 | raw 入库尚未完成时，消息放回订单缓冲等待下次处理 |

## 6. `t_order_raw` 原始报单入库筛选

入口：`OrderIngressService.batchIngest`

| 筛选顺序 | 规则 | 目的 |
| --- | --- | --- |
| 1 | source 为空时默认 `WECHAT`，非空转大写 | 统一来源格式 |
| 2 | `WECHAT`、`WECHAT_CALLBACK` 直接跳过 | 旧微信通道停用，防止旧回调混入 |
| 3 | 生成 raw fingerprint | 作为 raw 级消息身份 |
| 4 | 同一批次内相同 fingerprint 跳过 | 防止同批重复提交 |
| 5 | 同一批次跨来源镜像重复跳过 | 防止同一消息由多个通道同时进入 |
| 6 | 进入 `ingestLock` 同步区 | 防止高并发竞态 |
| 7 | 10 分钟 recent identity cache 命中跳过 | 防止刚写入的重复消息并发穿透 |
| 8 | DB 中 fingerprint 已存在跳过 | 防止历史重复 raw |
| 9 | DB 中跨来源镜像重复跳过 | 防止 Redis/SSE/回调镜像重复 |
| 10 | 插入 `t_order_raw` | 进入 AI 解析 |
| 11 | DB 唯一键异常兜底跳过 | 防止并发下最后一刻重复 |

raw fingerprint 生成规则：

| 情况 | seed |
| --- | --- |
| 有 `msgId` | `source|msgId` |
| 无 `msgId` | `source|fromWxid|senderWxid|UUID` |

跨来源镜像重复判断：

| 项目 | 规则 |
| --- | --- |
| 参与来源 | `WECHAT`、`WECHAT_REDIS`、`WECHAT_CALLBACK`、`WECHAT_SSE` |
| 旧来源处理 | DB 查询时排除 `WECHAT`、`WECHAT_CALLBACK`，避免旧停用来源压制 Redis 正常报单 |
| 判断字段 | `fromWxid`、`senderWxid`、`rawText`、`receivedAt` 完全一致 |
| 当前结论 | Redis 是当前有效来源，旧 WECHAT raw 保留历史但不应压制新数据 |

重要业务边界：

| 场景 | 处理方式 |
| --- | --- |
| 微信有效报单重复发送 | 不能仅因内容相同就丢弃 |
| 同一上游 `msgId` 重复推送 | 视为传输重复，跳过 |
| 不同消息 id 但文本相同 | 原则上保留，由后续解析和统计处理 |
| 旧 WECHAT 回调残留 | 当前不再作为有效来源参与统计新增 |

## 7. AI 解析筛选与纠偏

入口：`AsyncParseService`、`AiParseService`

| 环节 | 规则 |
| --- | --- |
| 解析方式 | 每条 `t_order_raw` 独立解析，避免长消息互相影响 |
| 实时并行 | `ai.realtime-parallel=16` |
| 重跑并行 | `ai.reparse-parallel=2` |
| AI 超时 | `ai.timeout=180000ms` |
| AI 重试 | 对 timeout、connection reset/refused、502、503、504 等重试 |
| AI call log | `ai.save-call-log=false` 时不保存调用日志 |

本地快路径优先于 AI API：

| 顺序 | 快路径 | 用途 |
| --- | --- | --- |
| 1 | exact prompt case | 命中历史人工纠偏精确案例 |
| 2 | simple direct list | 简单直选列表 |
| 3 | long half direct list | 长列表半数/金额类直选 |
| 4 | long direct list | 长直选列表 |
| 5 | annotated long permutation list | 带源注数标注的长复式/排列列表 |
| 6 | long permutation list | 长复式/排列列表 |

AI 提示词增强：

| 来源 | 代码 | 作用 |
| --- | --- | --- |
| 基础提示词 | `PromptService` | 彩票报单解析规则 |
| 目标群规则 | `TargetGroupRuleService` | “福停”后强制按体彩排列三处理 |
| 常见错字归一化 | `OrderTextNormalizationService` | 复试->复式、丹->单、租->组等 |
| 历史纠偏案例 | `PromptCaseService` | 将人工修正案例加入提示词或精确命中 |

AI 响应筛选：

| 条件 | 结果 |
| --- | --- |
| JSON 数组解析成功 | 进入结果纠偏 |
| JSON 为空 | 生成 `FAIL`，原因“AI响应中未找到任何结果” |
| JSON 解析异常 | 生成 `FAIL`，原因“响应解析失败” |
| AI 调用异常 | 生成 `FAIL`，记录异常信息 |
| `valid=true/status=SUCCESS` 且有 `data` | 作为成功候选 |
| `status=SKIP` | 作为跳过候选，不入订单明细 |
| `status=FAIL` | 作为失败候选，不入订单明细 |

主要后处理筛选/纠偏：

| 纠偏类型 | 作用 |
| --- | --- |
| exact case correction | 精确命中人工案例时优先采用案例结果 |
| 多行包票纠偏 | 按行解析复杂包票，防止 AI 合并错行 |
| 彩种作用域纠偏 | 多行、多段、空行中彩种只影响对应范围 |
| 直选/复式混合纠偏 | 一条消息同时存在直选和复式时分开处理 |
| 复式展开纠偏 | 将基础号码展开成全排列号码 |
| 共享注数纠偏 | 多行共用一个注数时补齐 |
| 金额保护 | 纠偏后总金额异常时回退可信基础结果 |
| 号码注数对纠偏 | 识别“号码 注数”成对结构 |
| 豹子/组选包票纠偏 | 特殊玩法结构化处理 |
| 两码跳过 | 仅两位号码且属于两码/二码玩法时标记 `SKIP` |
| 不支持组选跳过 | 明确属于暂不支持类型时标记 `SKIP` |
| 福停后强制体彩 | 目标群出现“福停”后，后续消息强制 `TC/P3` |

## 8. 解析结果落库筛选

入口：`NormalizePersistService.persistSingle`

| 顺序 | 规则 | 落表 |
| --- | --- | --- |
| 1 | 对 raw 执行 `selectByIdForUpdate` | 防止同一 raw 并发落库 |
| 2 | AI items 为空 | 保存失败 batch |
| 3 | 有成功 item | 如已有解析数据，先清理该 raw 历史解析和统计 |
| 4 | 无成功但有失败 | 保存失败 batch |
| 5 | 无成功但有跳过 | 保存跳过 batch |
| 6 | 保存 batch | `t_order_parse_batch` |
| 7 | 成功 batch 设 `is_effective=1` | 其他同 raw batch 置 `is_effective=0` |
| 8 | 每个 AI item 都保存记录 | `t_ai_parse_result` |
| 9 | 只有 success item 进入订单明细 | `t_order_item`、`t_order_item_number` |

batch 状态：

| 状态 | 数值 | 是否有效 | 是否进入统计 |
| --- | --- | --- | --- |
| 成功 | `parse_status=1` | `is_effective=1` | 是 |
| 部分成功 | `parse_status=2` | 当前常量保留 | 视具体实现 |
| 失败 | `parse_status=3` | `is_effective=0` | 否 |
| 跳过 | `parse_status=4` | `is_effective=0` | 否 |

## 9. 订单明细与号码统计筛选

入口：`NormalizePersistService.persistValidOrders`

| 筛选顺序 | 规则 | 结果 |
| --- | --- | --- |
| 1 | success item 的 `data` 为空 | 跳过该 item |
| 2 | `numbers` 为空 | 跳过该 item |
| 3 | 去掉 null 号码后为空 | 跳过该 item |
| 4 | `play` 为空 | 默认 `直选` |
| 5 | `zone` 为空 | 默认 `MAIN` |
| 6 | `bet` 非正数或为空 | 默认 `1` |
| 7 | `amount` 大于 0 | 按号码数量平均分摊到每个号码 |
| 8 | 同 raw 内贡献键重复 | 跳过重复贡献 |
| 9 | 有效号码为空 | 不生成 `order_item` |
| 10 | 批量保存订单项 | `t_order_item` |
| 11 | 批量保存号码明细 | `t_order_item_number` |
| 12 | 按统计键汇总后批量 upsert | 累加 `t_number_stats` |

同 raw 内统计防重键：

| 字段 | 说明 |
| --- | --- |
| `lotteryCategory` | 彩种 |
| `gameType` | 游戏 |
| `playType` | 玩法 |
| `numberZone` | 区域 |
| `number` | 号码 |
| `betCount` | 注数 |

该规则的目的不是删除重复微信报单，而是防止同一条 raw 被 AI 重复输出同一号码贡献，导致同一 raw 内重复加统计。

统计写入保护：

| 机制 | 说明 |
| --- | --- |
| `stat_applied` | 新落库路径在同一事务内写入 `1`，表示本 item 已随本次统计一起应用 |
| 同 raw 贡献键 | 入库前先去掉同 raw 内 AI 重复输出的同号码贡献 |
| 批量统计 upsert | 所有号码贡献先在内存按统计键汇总，再批量写入 `t_number_stats` |
| 事务保护 | 订单项、号码明细、统计任一失败会整体回滚 |
| `ON DUPLICATE KEY UPDATE` | 同号码维度存在则累加，不存在则插入 |

## 10. 清理与重跑规则

入口：`OrderReparseService`、`NormalizePersistService`

| 通道 | 是否自动 | 时间限制 | 说明 |
| --- | --- | --- | --- |
| 自动补偿重跑 | 当前关闭 | 受窗口限制 | `order.auto-reparse-enabled=false` |
| `/api/reparse` | 手动 | 受窗口限制 | 普通手动按日期重跑 |
| `/api/force-parse` | 手动 | 不受窗口限制 | 强制按日期重跑 |
| `/api/force-parse-fast` | 手动 | 不受窗口限制 | 使用实时 lane 快速重跑 |
| `/api/force-parse-raw` | 手动 | 不受窗口限制 | 单 raw 重跑 |
| `/api/reparse-corrected-raw` | 手动 | 不受窗口限制 | 带人工校正说明重跑 |

重跑候选筛选：

| 条件 | 规则 |
| --- | --- |
| 成功 batch | 不重跑 |
| 失败 batch | 只有 parseMsg 像 AI API、timeout、无法解析等可重试错误才重跑 |
| 无 batch | 可能实时解析仍在路上，需满足更长冷却 |
| 重试次数 | 达到 `order.reparse-max-attempts` 后跳过 |
| 冷却时间 | 默认 `order.reparse-cooldown-seconds` |
| 批大小 | 最多 `order.reparse-batch-size` |
| 并发保护 | `retryRunning` 防止多个补偿任务同时跑 |

清理逻辑：

| 清理动作 | 说明 |
| --- | --- |
| 锁定 raw | `selectByIdForUpdate` 防止并发清理/落库 |
| 查旧 batch | 包含历史 batch id |
| 查旧 item | 按 raw_id 和 batch_id 双路径查，兜底错挂明细 |
| 回滚统计 | 仅对 `stat_applied=1` 的 item 扣减统计 |
| 删除号码明细 | 删除 `t_order_item_number` |
| 删除订单项 | 删除 `t_order_item` |
| 删除 parse batch | 删除 `t_order_parse_batch` |
| 删除 AI 结果 | 删除 `t_ai_parse_result` |
| 日期重跑可清空统计 | 传入 issueKey 时删除该日期 `t_number_stats` 后重建 |
| 删除非正统计 | 清除 `order_count<=0` 或 `sum_amount<=0` 的统计行 |

## 11. 查询展示筛选

入口：`StatsController`、`StatsService`

| 接口 | 作用 |
| --- | --- |
| `GET /wechat/api/check-messages` | 检查今天 19:30 后 DB 消息数和 Redis 队列数 |
| `GET /wechat/check/messages` | 兼容消息检查接口 |
| `GET /wechat/api/runtime-status` | 查看缓冲、raw、成功、失败、跳过数量 |
| `GET /wechat/api/stats` | 查询号码统计 |
| `GET /wechat/api/top-source-messages` | 按号码或 raw 查询来源消息 |
| `POST /wechat/api/delete-date-data` | 清理指定日期解析结果，保留 raw |
| `POST /wechat/api/force-parse*` | 强制重跑 |

统计查询规则：

| 筛选项 | 说明 |
| --- | --- |
| 必填 | `lotteryCategory`、`gameType`、`issueKey` |
| 可选 | `playType`、`numberZone`、`number` |
| 排序 | `count` 对应注数，`amount` 对应金额 |
| 分页 | `page`、`pageSize` |
| 最大页大小 | `stats.max-top-n=500` |
| 数据来源优先级 | 优先读 `t_number_stats`，必要时从最新成功 AI 结果回退计算 |

来源消息查询规则：

| 筛选项 | 说明 |
| --- | --- |
| `rawId` | 指定单条 raw |
| `betNumber` | 只返回包含该号码的成功 raw |
| `messageKeyword` | raw 原文包含关键词 |
| 返回内容 | rawId、receivedAt、rawText、解析记录、总注数、总金额 |

## 12. 核查用 SQL 思路

以下是排查数据链路时常用的核查方向，实际执行时按目标日期替换 `:date`。

| 目标 | 核查点 |
| --- | --- |
| Redis 是否积压 | `LLEN wechat_messages` 或 `/wechat/api/check-messages` |
| raw 是否都解析 | `t_order_raw left join t_order_parse_batch` 找无 batch raw |
| 最新状态分布 | 每个 raw 取最新 batch，按 `parse_status` 分组 |
| 是否有失败 | 最新 batch `parse_status=3` |
| 统计是否一致 | 从 `t_order_item + t_order_item_number + effective batch` 重算，对比 `t_number_stats` |
| 是否有无效 batch 入统计 | `t_order_item` 关联 batch 非成功或非 effective |
| 是否有空号码成功项 | 成功有效 item 但无 `t_order_item_number` |
| raw 层重复 | 按 source/from/sender/rawText/receivedAt 分组 |
| 近似重复 | 同 source/from/sender/rawText 在 60 秒内成对出现 |

raw 层重复不一定是错误。判断原则是：

| 类型 | 是否错误 | 处理 |
| --- | --- | --- |
| 同一上游 msgId 重复 | 通常是传输重复 | 应被 `t_message` 或 `t_order_raw` 去重 |
| 不同上游消息但文本相同 | 可能是有效重复报单 | 不应按内容硬删除 |
| 旧 WECHAT 与 WECHAT_REDIS 镜像 | 当前旧通道停用后不应生效 | 应跳过旧来源，保留 Redis |
| 同 raw 内 AI 重复输出 | 通常是解析冗余 | 由贡献键和 `stat_applied` 防重 |

## 13. 当前重点保护规则

| 风险 | 当前保护 |
| --- | --- |
| 旧千寻回调混入 | `/callback` 停用，旧 `WECHAT`/`WECHAT_CALLBACK` raw 源在新入库阶段跳过 |
| Redis 高峰积压 | 订单缓冲 50 条或 500ms 即刷写，实时 AI 并行 16 |
| 自动重跑抢占高峰资源 | 自动补偿重跑关闭，手动重跑保留 |
| 相同文本有效报单被吞 | 无上游 id 时引入 UUID，不按内容生成唯一身份 |
| 跨来源镜像重复 | 同 source 以外完全同 raw 身份会跳过，但排除停用旧来源 |
| AI 重复输出同号码 | 同 raw 内贡献键防重，`stat_applied` 防重复统计 |
| 单 raw 重跑重复入账 | 重跑前先清理旧明细并扣减旧统计 |
| 长复式/排列单 AI 超时 | 本地长列表快路径优先处理 |

## 14. 业务排障路径

| 问题 | 优先检查 |
| --- | --- |
| 今天是否还在收消息 | `/wechat/api/check-messages`、Redis 队列长度、`t_message` 最新时间 |
| 消息是否进入 raw | `t_message` 与 `t_order_raw` 按时间、sender、内容对比 |
| raw 是否漏解析 | raw 左连接 batch 找 `batch_id is null` |
| 解析是否失败 | 最新 batch `parse_status=3` 和 `parse_msg` |
| 注数偏高 | 查 `t_order_item`、`t_order_item_number`、同 raw 贡献重复、raw 层重复 |
| 某号码来源 | `/wechat/api/top-source-messages?betNumber=xxx` |
| 某 raw 明细 | 查 `t_order_parse_batch`、`t_ai_parse_result`、`t_order_item`、`t_order_item_number` |
| 统计不一致 | 用明细重算并对比 `t_number_stats` |
| 旧通道影响 | 按 `source` 分组检查 `t_order_raw` 和有效 batch |

## 15. 可优化点

以下优化点按“准确性优先、再吞吐、再观测”的顺序排列。2026-05-06 已落实第一批高优先级优化。

| 优先级 | 优化点 | 落实状态 | 当前实现 | 预期收益 |
| --- | --- | --- | --- | --- |
| P0 | Redis 消息可靠出队 | 已落实 | `RedisClientService` 使用 `rightPopAndLeftPush` 先移入 `wechat_messages:processing`，处理成功后 ack；启动时可恢复 processing 队列 | 避免消息已出 Redis 但服务宕机导致丢消息 |
| P0 | Redis 重放断点续跑 | 已落实 | 如果 Redis 重放时 `t_message` 已存在，仍继续进入订单 raw 检查；如果 raw 已存在但无 batch，则返回 existing raw 继续解析 | 覆盖“消息已入库但 raw/解析未完成”这类重启断点 |
| P0 | raw 入库串行瓶颈 | 已部分落实 | 同批 fingerprint 先批量预查，减少逐条 DB 查询；跨来源镜像 DB 检查默认关闭，可配置开启 | 高峰 raw 入库 DB 往返减少 |
| P0 | 大单落库事务过长 | 已落实 | `t_ai_parse_result`、`t_order_item`、`t_order_item_number` 批量 insert；`t_number_stats` 先聚合再批量 upsert | 大幅降低长复式/排列单的事务耗时和连接占用 |
| P1 | Redis 消费吞吐 | 已落实 | 增加 `redis.consumer-threads`，当前配置为 4 个消费线程 | 高峰突发消息更快进入 `t_message` 和订单缓冲 |
| P1 | `t_message` 入库去重查询 | 暂未改 | 当前仍是先查重复再 insert，保持保守模式 | 后续确认历史唯一索引完整后可改 insert-first |
| P1 | 解析队列可观测性 | 已落实 | `/runtime-status` 增加 Redis 主队列/processing、异步 parse executor、AI realtime/reparse executor、rawWithoutAnyBatchCount | 堵塞时能快速判断卡在 Redis、raw、AI 还是落库 |
| P1 | 高峰资源隔离 | 已落实 | `force-parse-fast` 默认在实时收单窗口禁用，普通重跑仍走 REPARSE lane | 防止人工快速回放抢占实时解析资源 |
| P2 | 文档中的配置值维护 | 部分落实 | 本节记录已落地配置，具体仍以 `application.yml` 为准 | 降低文档和真实配置不一致风险 |
| P2 | AI 纠偏策略沉淀 | 持续进行 | 继续通过 prompt case 和提示词沉淀口语表达；本次未新增硬编码解析规则 | 在尽量少硬编码的前提下提高解析准确率 |
