# 群消息批量处理方案

## 需求概述
- 只接受群wxid为 `45995021836` 的消息
- 每30秒将收到的消息统一批量处理
- 不能漏掉任何消息，也不能重复保存
- 存入 `E:\qian-xun-pro-wechat-http-demo-master\src\main\java\cn\daenx\myadmin\data\msg.txt`

## 技术方案

### 核心组件

#### 1. MessageBufferService（消息缓冲服务）

**职责**：
- 线程安全地存储待处理消息
- 使用 `msgId` 去重，防止重复保存
- 提供原子性的"获取并清空"操作
- **写入失败时支持消息恢复**

**接口定义**：
```java
public interface MessageBufferService {

    /**
     * 添加消息到缓冲区
     */
    void addMessage(RecvMsgReqVo msg);

    /**
     * 原子性获取所有消息并清空缓冲区
     */
    Map<String, RecvMsgReqVo> flushAndClear();

    /**
     * 写入失败时，将消息恢复到缓冲区
     */
    void restoreMessages(Map<String, RecvMsgReqVo> messages);
}
```

**关键设计**：
```
存储结构：ConcurrentHashMap<String, RecvMsgReqVo>
  - Key: msgId（消息唯一标识）
  - Value: 消息对象

方法：
  - addMessage(RecvMsgReqVo msg): 添加消息到缓冲区
  - flushAndClear(): 原子性获取所有消息并清空缓冲区
  - restoreMessages(Map): 写入失败时恢复消息到缓冲区
```

**为什么用 ConcurrentHashMap**：
- 线程安全，支持高并发写入
- 天然去重（相同msgId会覆盖）
- 读写性能优秀

**原子性 Swap 实现（重要）**：

普通的 `clear()` + `values()` 不是原子操作，会导致消息丢失。采用 **Swap 模式** 实现真正的原子性：

```java
@Service
public class MessageBufferServiceImpl implements MessageBufferService {

    private volatile ConcurrentHashMap<String, RecvMsgReqVo> buffer = new ConcurrentHashMap<>();
    private final Object swapLock = new Object();

    @Override
    public void addMessage(RecvMsgReqVo msg) {
        buffer.put(msg.getMsgId(), msg);
    }

    @Override
    public Map<String, RecvMsgReqVo> flushAndClear() {
        synchronized (swapLock) {
            Map<String, RecvMsgReqVo> snapshot = buffer;
            buffer = new ConcurrentHashMap<>();
            return snapshot;
        }
    }

    @Override
    public void restoreMessages(Map<String, RecvMsgReqVo> messages) {
        // 写入失败时，将消息放回缓冲区（使用 putIfAbsent 避免覆盖新消息）
        messages.forEach((msgId, msg) -> buffer.putIfAbsent(msgId, msg));
    }
}
```

**Swap 模式原理**：
1. 用 `synchronized` 保证 swap 操作的原子性
2. 直接交换引用，而非清空原 Map
3. 新消息写入新的 Map，不影响正在处理的旧数据
4. `volatile` 保证多线程可见性

#### 2. MessageFlushTask（定时刷写任务）

**职责**：
- 每30秒执行一次
- 从缓冲区获取所有消息
- 追加写入到txt文件
- **写入失败时自动恢复消息，下次重试**

**关键设计（含失败恢复机制）**：
```java
@Component
@Slf4j
public class MessageFlushTask {

    @Autowired
    private MessageBufferService bufferService;

    private static final String FILE_PATH = "E:\\qian-xun-pro-wechat-http-demo-master\\src\\main\\java\\cn\\daenx\\myadmin\\data\\msg.txt";

    @Scheduled(fixedRate = 30000)  // 每30秒执行
    public void flush() {
        // 1. 原子性获取并清空缓冲区
        Map<String, RecvMsgReqVo> messages = bufferService.flushAndClear();

        if (messages.isEmpty()) {
            return;
        }

        log.info("开始刷写消息，数量: {}", messages.size());

        try {
            // 2. 写入文件
            writeToFile(messages.values());
            log.info("消息刷写成功，数量: {}", messages.size());

        } catch (IOException e) {
            // 3. 写入失败，消息放回缓冲区等待下次重试
            log.error("消息写入文件失败，已放回缓冲区等待重试", e);
            bufferService.restoreMessages(messages);
        }
    }

    private void writeToFile(Collection<RecvMsgReqVo> messages) throws IOException {
        try (BufferedWriter writer = new BufferedWriter(
                new FileWriter(FILE_PATH, StandardCharsets.UTF_8, true))) {

            for (RecvMsgReqVo msg : messages) {
                String json = String.format("{\"msg\":\"%s\",\"timeStamp\":\"%s\"}",
                        escapeJson(msg.getMsg()),
                        msg.getTimeStamp());
                writer.write(json);
                writer.newLine();
            }
            writer.flush();
        }
    }

    private String escapeJson(String str) {
        if (str == null) return "";
        return str.replace("\\", "\\\\")
                  .replace("\"", "\\\"")
                  .replace("\n", "\\n")
                  .replace("\r", "\\r");
    }
}
```

**失败恢复机制说明**：
1. 写入失败时，调用 `restoreMessages()` 将消息放回缓冲区
2. 使用 `putIfAbsent` 避免覆盖失败期间新到达的消息
3. 下一个30秒周期会自动重试
4. 消息不会丢失，只会延迟写入

### 防止消息丢失/重复的机制

| 风险 | 解决方案 |
|-----|---------|
| 并发写入冲突 | 使用 ConcurrentHashMap 保证线程安全 |
| 消息重复 | 使用 msgId 作为 key，自动去重 |
| 定时任务执行时新消息到达 | **采用 Swap 模式**：用 synchronized + volatile 实现真正的原子交换，新消息写入新 Map |
| **文件写入失败** | **失败恢复机制**：写入异常时调用 `restoreMessages()` 将消息放回缓冲区，下次定时任务自动重试 |
| 应用重启导致缓冲区丢失 | 这是个风险点，如需解决可考虑持久化方案 |
